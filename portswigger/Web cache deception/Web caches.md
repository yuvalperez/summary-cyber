
(עשיתי # מה זה cahce) 
אז מה זה web cache זאת מערכת שיושבת בין השרת המקור לבית המשתמש.
מתי שהמתשמש מבקש משאב סטטי (משאב סטטי = קובץ שלא משתנה לפי המשתמש) כמו לוגו, קבצי CSS של עציבוב  קבצי java  וכו. הבקשה מופנת קודם אל הcache אם לcache אין את העותק הנדרש יודע כcaches miss , הבקשה תופנה לשרת השרת שולח את העותק לcache ואז הcache למשתמש . לcache יש כלים כדאי לדעת עם הוא צריך לשמור ולאחסן את התגובה או לא.
מתי שנשלח אץ אותו בקשה סטטית בעתיד  הcahce שולח את הבקשה המאוחסנת של התגובה ישר למשתמש (יודע כcahce hit ).

חידוד:
א. cache miss = ה cache לא מכיר את הבקשה → שולח לשרת המקור → שרת מחזיר תגובה ל cache → cache שומר ושולח למשתמש.
ב. cache hit = ה cache מכיר את הבקשה → שולח ישר למשתמש בלי לגעת בשרת.


## Cache keys: #Cache-keys

אז מתי שהcahce מקבל בקשה http  הוא מחויב להחליט אם  יש ילו תגובה לתת או שהוא מעביר את הבקשה לשרת , הcache מקבל החלטה לידי יצירת מפתח (Cache keys) מאלמנטים של הבקשה http. בדרך כלל זה כולל URL ופרמטרים query  אבל זה גם יכול להכיל עוד דברים כמו אלמנטים אחרים ,כותרות וסוג תוכן. אם הבקשה תואמת את הcache key מהבקשה הקודמת אז הcache מחשיב את הבקשה לאותו בקשה כמו הקודמת ומביא עותק למשתמש. 

מושגים:
א. החלק שהוא בודק (cache key).
ב. החלק שהוא לא  בודק (unkeyed).

אם הcache בודק  url אבל לא headers  הcache חושב שזאת אותה בקשה אבל הבעיה היא שזאת בקשה אחרת לגמרי ומסוכנת , למה? הcahce מתעלם מהheader והתוקף יכול להכניס תוכן זדוני והcache שומר  את זה . נקראה web cache poisoning. 

## Cache rules: #Cache-rules
חוקי הCache מחליטים מה יכול להיות cached ולאורך כמה זמן. בדרך כלל החוקים הם סטטים שכמעט ולא משתנים ותואמים לכמה עמודים של האתר. דברים דינמים בדרך לא נכנסים לcache בלל שזה יכול להכיל מידע רגיש כדאי שהמשתמש יקבל את הנתונים העדכניים ביותר ישרות מהשרת. התקפות cache deception (הטעיית מטמון אינטרנט) בדרך כלל נגרם בגלל החוקים של הcache, אז זה חשוב לדעת על כל מיני חוקים שונים, במיוחד האלו שמבוססים עלURL שמוחלט על ידי  מחרוזת  (strings ).
לדוגמה: 
* כללי סיומת קבצים סטטיים (Static file extension rules) - כללים אלו תואמים את סיומת של המשאב המבוקש , למשל stylesheets.css וכו.
* כללי ספריות סטטיים (Static directory rules) - כללים אלה בודקים אם את כל נתבי URL מתחילים בקידומות ספציפיות. 
* כללי שם קבצים (File name rules)- החוק בודק רק את השם של הקובץ.

הcache יכול גם להיות עם כללים המבוססים קיטריונים אחרים. 


## Constructing a web cache deception attack: #-Constructing-a-web-cache-deception-attack

בניית מתקפת הטעיה באמצעות מטמון אינטרנט (Constructing a web cache deception attack)
בדרך כלל בשלבים היו ככה
לזהות endpoint (נקודת קצה) שמחזירה תגובה דינמית שמכילה מידע רגיש. לרואת את הבקשה בburp יכול להיות בגלל שהמידע רגיש הוא לא היה גלוי בדף.  צריך להתמקד ב endpoint שתומך בGET,HEADA או options  בגלל שהבקשות משנות את מצב של השרת המקורי בדרך כלל לא היו מאוחסנות בcahce. 
זיהוי פער באופן שבו cache ושרת המקור מנתחים את נתיב ה-URL. זה יכול להיות פער באופן שבו הם:
מפים כתובות URL למשאבים. מעבדים תווי הפרדה. מנרמלים נתיבים.
יוצרים URL המשמש לעבוד על הcahce ולהחסן בקשות דינמיות ומתי שהקורבן נכנס לurl  הבקשה שלהם מאוחסנת בcahce ובאמצעות Burp אפשר לשלוח בקשה שלוקחת התגובה המאוחסנת הcahce ולקבל את הפרטים של הקורבן. חשוב להמנע מלעשות את זה בדפדפן בגלל מגבלות של האפלקציה שיכולה לטשטש את הפריצה.

## Using a cache buster: #Using-a-cache-buster 
חשוב שבזמן שבודקים פערים ומכינים web cache deception exploit  לבוודא שכל בקשה שאנחנו עושים תהיה עם key cache  אחר. אחרת יכול להיות שנקבל תשובות שכבר שמורות בcache וזה ישפע על התוצאות שלנו.
בגלל שהURL path  וגם query parameters בדרך כלל כלולים בcache key, אפשר לשנות מפתח על ידי query string וpath ולשונת כל פעם ששולחים בקשה . כדאי להפוך את התהליך לאוטומטי נשתמש בתוסף  Param Miner.

## Detecting cached responses: #-Detecting-cached-responses
שאנחנו בודקים זה חשוב מאוד שנדע להבדיל בין תגובות. בשביל לעשות את זה אנחנו נתסכל על כותרות (headers) ועל זמן תגובה. 

כותרת ה `x-cache`  מכילה מידע על התגובה שהגיע מהcahce..
* כותרת `x-cahce:hit` - מכילה  את תגובה מהcahce.
* כותרת `x-cahce:miss` - הcache לא הכיל את התגובה של אותו מפתח והתגובה נלקחה מהשרת , התגובה נשמרת בcache אם נשלח את אותה בקשה נקבל מהcache .
* כותרת `X-Cache: dynamic` - זה אומר שהבקשה נוצרה מהשרת והcache לא אחראי עליה .
* כותרת `X-Cache: refresh` - המידע צריך רענון או אימות חוזר.
* כותרת `Cache-Control` - מתי שהשרת מוסיף כותרת  כזאת הוא אומר לcahce מה לעשות עם התגובה .
  למשל:  Cache-Control: public, max-age=3600
זה אומר שpublic כולם יכולים לעשות לזה cache לא רק התדפדפן 
וmax-age=3600 כמה שניות לשמור את זה.

עוד נקודה בגלל שהשרת שלח Cache-Control: public לא אומר שהcache חייב לשמור את זה.


## Exploiting static extension cache rules : #Exploiting-static-extension-cache-rules

בדרך כלל חוקי הcache מכוונים משאבים סטטים  על ידי סיומת קבצים כמו .css וכו. זאת התנהגות מוכרת ברוב CDNs (עשיתי # מה זה ).
עם יש פער בין הcache לבין השרת המקורי. בין החיבורים של המשאבים, התוקף אולי יוכל להכין בקשה למשאב הדינמי עם הסיומת הסטטית שהשרת מתעלם ממנה אבל הcahce מקבל אותה. 
דוגמה:
תוקף שולח: /profile/test.css
השרת: מתעלם מ .css → מחזיר פרטי פרופיל דינמיים
ה cache: רואה .css → חושב שזה סטטי → שומר את זה

עכשיו כל אחד שמבקש /profile/test.css
מקבל את פרטי הפרופיל של הקורבן מה cache


חידוד:
### הבדל חשוב  (Poisoning vs Deception):

- א. **Web Cache Poisoning:** התוקף שולח בקשה "מורעלת" $\rightarrow$ ה-Cache שומר אותה $\rightarrow$ **כל המשתמשים** נפגעים (למשל מקבלים JavaScript זדוני).

- ב**Web Cache Deception:** התוקף גורם **לקורבן** לבקש URL מיוחד $\rightarrow$ המידע הפרטי של הקורבן נשמר ב-Cache $\rightarrow$ **התוקף** ניגש ל-URL וגונב את המידע.

עוד כמה נקודות

- א. **של הגנה Headers :**  המתקפה: `Cache-Control: no-store`. זה אומר ל-Cache: "אל תשמור את זה בחיים, גם אם זה נראה לך סטטי".
    
- ב. **Path Mapping:** לפעמים השרת משתמש ב-**Path Mapping** שונה. לדוגמה, אם תכתוב `/profile/nonexistent.css`, השרת עשוי להחזיר את `/profile` כי הוא פשוט מתעלם ממה שבא אחרי הסלאש האחרון.
