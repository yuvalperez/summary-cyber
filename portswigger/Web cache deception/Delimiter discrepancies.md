פערי תוחם(Delimiter discrepancies) תווי תוחם מציינים גבולות בין אלמנטים שונים בכתובות URL. השימוש בתווים (characters) ובמחרוזות (strings) כמפרידים הוא בדרך כלל סטנדרטי. 
למשל סימן שאלה נועד כדאי להפריד בין הURL לquery string. פערים באיך שהשרת והcache משתמשים  בתווים(characters) ומחרוזות (strings)  גורמות לפגיעות (web cache deception vulnerabilities ) .
למשל: /profile;foo.css 
המערכת Java Spring משתמשת ב ; כדאי להוסיף פרמטרים שקוראים (matrix variables) ובגלל זה שרת שמשתמש במערכת הזאת חותך את הנתיב אחרי  `/profile` ומחזיר מידע על הפרופיל. 

רוב המערכות אחרות לא משתמשות ב ; כתו מפריד. לכן הcache שלא משתמש במערכת Java Spring יפרש את כל מה שאחרי ;  כחלק מהנתיב(path) אז עם לשרת יש חוק שאומר שהוא שומר בקשות שמסתיימות בcss. אז הcahce ישמור את הבקשה והופ יש לנו פריצה.

הפער נובע מכך שאין **RFC** אחד שכולם מצייתים לו ב-100% לגבי תווים מיוחדים בנתיב. כל טכנולוגיה (Spring, Django, ASP.NET) מימשה את זה קצת אחרת.

מפריד `.` (Format Delimiter) 
 
  נפוץ ב-Ruby on Rails השרת משתמש בנקודה כדי להחליט באיזה פורמט להחזיר את הדף. אם הוא לא מכיר את הסיומת, הוא פשוט מתעלם ממנה ומחזיר את ברירת המחדל (HTML).
- **דוגמה:** `/profile.ico`
- **בשרת:** לא מכיר פורמט `.ico` לפרופיל, אז הוא מביא את דף ה-HTML הרגיל.
- **ב-Cache:** רואה סיומת של אייקון (`.ico`), בטוח שזה קובץ תמונה ושומר אותו אצלו.


**איך בודקים את זה?** מוסיפים `;test.css` או `.ico` לסוף ה-URL של דף עם מידע רגיש. אם הדף נטען רגיל וב-Headers מופיע `X-Cache: HIT`, מצאת פרצה.

### תווי תוחם מקודדים (Encoded Delimiters)

לפעמים השרת וה-Cache לא מסכימים על תווים שהם "בלתי נראים" או מקודדים, כמו ה-**Null Byte** (`%00`).

**השימוש ב-Null Byte (`%00`) - נפוץ בשרתי OpenLiteSpeed** שרת OpenLiteSpeed מתייחס לתו `%00` בתור סוף המחרוזת (Delimiter). הוא פשוט מפסיק לקרוא את הכתובת באותה נקודה.

- **דוגמה:** `/profile%00foo.js`
- **בשרת (OpenLiteSpeed):** רואה את ה-`%00`, עוצר שם, ומגיש את הדף `/profile` (המידע הפרטי).
- **ב-Cache (כמו Akamai או Fastly):** לא עוצר ב-`%00`. הוא רואה את כל הכתובת כנתיב אחד ארוך שמסתיים ב-`.js`.

- **התוצאה:** ה-Cache חושב שמדובר בקובץ JavaScript סטטי ושומר את תוכן הפרופיל הפרטי בזיכרון שלו.


**איך בודקים את זה?** מנסים להזריק `%00` ואחריו סיומת סטטית (כמו `.js` או `.png`) לסוף ה-URL.

1. אם האתר עדיין מציג את הדף המקורי – השרת "חתך" את הכתובת בגלל ה-Null Byte.
2. אם אחרי רענון מופיע `X-Cache: HIT` – ה-Cache "נפל בפח" ושמר את הדף.

**איך מוצאים את ה"פרצה"** 
כדי לנצל פער, אנחנו צריכים למצוא תו שהשרת מתייחס אליו כסוף הכתובת (Delimiter), אבל ה-Cache חושב שהוא סתם חלק מהשם.

**שלב 1: יצירת נקודת ייחוס (Reference)** נוסיף סתם אותיות לסוף ה-URL כדי לראות איך השרת מגיב לנתיב שלא קיים.

- **דוגמה:** משנים את `/profile` ל-`/profileaaa`.

- **התוצאה הצפויה:** שגיאת 404 או דף לא נמצא. זה ה"סימן" שלנו לנתיב שבור.

**שלב 2: בדיקת תו התוחם (The Test)** עכשיו נכניס תו חשוד (כמו `;` או `.`) בין הנתיב המקורי לבין האותיות שהוספנו.

- **בדיקה:** `/profile;aaa`

**שלב 3: ניתוח התוצאות** נשווה את התגובה שקיבלנו למה שראינו בשלב 1:

- **השרת חתך (בינגו!):** אם קיבלת את דף הפרופיל התקין, זה אומר שהשרת ראה את ה-`;`, עצר שם והתעלם מכל מה שבא אחריו. מצאת תוחם!

- **השרת לא חתך:** אם קיבלת שוב 404 (כמו ב-`/profileaaa`), זה אומר שהשרת חושב שה-`;` הוא חלק מהשם של הדף, ולכן הוא לא תוחם בשבילו.

**למה זה עוזר לנו?** אם מצאת שהשרת חותך ב-`;`, עכשיו אנחנו יכולים להוסיף סיומת סטטית שה-Cache אוהב: `/profile;aaa.css` השרת יביא את הפרופיל, אבל ה-Cache יראה `.css` וישמור את הדף.

### בדיקת פערים מול ה-Cache (The Final Test)

אחרי שזיהית שהשרת חותך את הנתיב בתו מסוים (למשל `;`), אתה צריך לוודא שה-Cache לא חותך שם, אלא ממשיך לקרוא עד לסיומת הסטטית.

**איך בודקים בפועל?** מוסיפים את התוחם שמצאת ואחריו סיומת של קובץ סטטי שה-Cache בדרך כלל שומר (כמו `.js`, `.css`, או `.ico`).

- **דוגמה:** `/settings/users/list;aaa.js`

**פירוח התוצאות:**

1. אם השרת מחזיר את הדף התקין (ה-List) – סימן שהתוחם עובד בצד השרת.

2. אם ב-Response מופיע Header של `X-Cache: HIT` (או משהו דומה) – סימן שה-Cache לא השתמש בתוחם, ראה סיומת `.js` ושמר את הדף הפרטי אצלו.

### בניית ה-Exploit והזרקת ה-Payload

אחרי שמצאנו את התו שמפריד (Delimiter), אנחנו בונים את הכתובת שתגרום ל-Cache לשמור את המידע הרגיש.

**דוגמה ל-Payload סופי:** `/settings/users/list;aaa.js`

**מה קורה כאן?**

1. **ה-Cache:** רואה את הכתובת המלאה, מזהה סיומת `.js` ושומר את התגובה בזיכרון (כי יש לו חוק שאומר "תשמור קבצי JS").
2. **השרת (Origin):** רואה את ה-`;`, עוצר שם ומגיש את הדף `/settings/users/list` (המידע הפרטי של המשתמש).
3. **התוצאה:** המידע הפרטי של המשתמש "נלכד" בתוך קובץ דמה ב-Cache.



### המגבלות של הדפדפן (Browser-Side Processing)

צריך לזכור שהבקשה עוברת דרך הדפדפן של הקורבן לפני שהיא מגיעה ל-Cache, ויש תווים שהדפדפן "מטפל" בהם בעצמו:

- **סימן `#` (Fragment):** הדפדפן חותך את כל מה שבא אחריו ולא שולח את זה לשרת בכלל. לכן, אי אפשר להשתמש ב-`#` בתור תוחם ב-Exploit ישיר.

- **תווים מיוחדים (`<`, `>`, `{`, `}`):** דפדפנים מודרניים מקודדים אותם אוטומטית (URL Encoding).

**הפתרון:** אם השרת או ה-Cache יודעים לפענח (Decode) את התווים האלה, אפשר לנסות להשתמש בגרסה המקודדת שלהם בתוך ה-Payload (למשל `%23` במקום `#`).


###  טיפ לסיכום (Scale):

מכיוון ששרתים בדרך כלל משתמשים באותם תוחמים לכל אורך המערכת, ברגע שמצאת תו אחד שעובד (כמו `;` ב-Java Spring), אתה יכול להשליך את זה על **כל שאר הנתיבים** באתר ולמצוא המון מידע רגיש.
